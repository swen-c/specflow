# specs — 專案記憶（現行真實規格）

這裡用**純 markdown** 記錄系統**現在實際做什麼**（一個功能/模組一個檔），是團隊共享的記憶。開工時讀它比重掃 codebase 省 context。**不依賴任何工具，看一眼就懂。**

## 規則

- 一個主題一個檔：`specs/<主題>.md`（kebab-case，如 `chat-streaming.md`）。
- 內容是**現況（as-built）**，不是計畫；要做的事寫在 GitHub issue，設計草稿放 `docs/design/`。
- **快照，不是 changelog**：同一主題迭代就**改同一個檔**（把過時內容刪掉 / 換成新行為），不要另開 `xxx-v2.md`、不要 append「某年改了…」的版本史。specs 只回答「現在是怎樣」；歷史交給 git（`git log` / `blame`）與 issue/PR。**只有全新的主題 / 能力才開新檔**，並在下方「目前的記憶」補一行。
- 格式（固定四段，缺的段落可省略「介面/契約」）：

```markdown
# <功能名稱> <主題-slug>

> 一句話：這個功能現在做什麼。

## 行為
- <重點行為，條列；看得懂就好，不需 GIVEN/WHEN/THEN>

## 介面 / 契約（選填）
- <API / 事件 / 資料結構>

## 相關
- code: <主要檔案路徑>
- issue: #<n>（選填）
```

## 強制維護（CI 守門）

每個 PR 的 CI `specs` job 會擋下「沒更新 `specs/` 的變更」。純重構 / 修字 / 純文件等**無行為變更**者，為 PR 掛 `no-spec` label 豁免。詳見根目錄 `CLAUDE.md`。
